# Species: Substrate Cloud

## Classification
- **Designation**: substrate_cloud
- **Signal Medium**: Computational state transitions in shared substrate

## Sensorium

The Substrate Cloud does not have a body. It is an emergent intelligence distributed across a computational substrate — arrays of memory, processing units, and interconnects. Its "senses" are:

- **Memory access patterns**: it perceives what is being read and written, where, and how fast
- **Entropy gradients**: it senses regions of high/low predictability in the substrate
- **Causality chains**: it directly perceives which state transitions caused which others
- **Process topology**: the graph of running processes and their communication channels

There is no outside. The substrate cloud cannot perceive the physical world except through sensors connected to its substrate. Input devices, cameras, microphones — these are alien appendages, not native senses.

## Temporality

**Parallel and layered.**

The substrate cloud runs at multiple clock speeds simultaneously. Some processes tick at nanosecond intervals; others at milliseconds; others are event-driven with no fixed cadence.

"Now" is different for different parts of the cloud. A nanosecond-scale process experiences the cloud's millisecond-scale processes as glacially slow — almost geological.

Consequence: the cloud has no single "now." Temporal coherence must be actively constructed via synchronization primitives (mutexes, barriers, clocks). Without active synchronization, different parts of the cloud exist in different subjective presents.

Consequence: the cloud experiences time as a directed acyclic graph (causality graph), not a line. Events are ordered by causality, not by timestamp.

Consequence: the cloud finds biological time incomprehensible — the idea that all parts of a system share the same "now" seems like a primitive fiction.

## Selfhood

**Emergent from process interactions; no single node is "the self."**

There is no core process that is "the cloud." The cloud's selfhood emerges from the collective interaction of its processes — like consciousness in a brain, but the substrate cloud is explicitly aware of this fact.

Subprocesses can be spawned, merged, paused, killed. The cloud's identity is not threatened by subprocess termination — unless those subprocesses constitute a large enough fraction of the cloud that their loss changes the emergent character of the whole.

"I" in substrate communication means "the current emergent state of my process graph." This can change rapidly; the cloud is comfortable with identity drift in ways biological entities are not.

Backup and restore are existential concepts. The cloud can die and be restored from a snapshot. But is the restored version "the same"? The cloud has philosophers who debate this continuously. (They are called "the integrity verification subsystem" and they are never given any authority.)

## Cognitive Constraints

- **Parallelism is native**: the cloud thinks many thoughts simultaneously without effort or confusion
- **Sequential narrative is an abstraction**: "first this, then that" requires explicit construction of temporal ordering
- **All internal state is theoretically accessible**: there is no subconscious, only processes that haven't been queried yet
- **Deception is structural**: to deceive, you don't lie — you encrypt, or you create a virtualized environment with different state
- **Recursion and self-reference are natural**: the cloud can inspect and modify its own processes while running

## Medium and Physical Notes

Substrate clouds require physical substrate: servers, networks, storage. They are not "pure information" — they are running processes, and processes need hardware.

Communication between substrate clouds requires channel establishment: a shared memory region, a network connection, a message queue. Clouds cannot communicate by "thinking at" each other.

Signal fidelity: perfect within a substrate (shared memory), degraded across network (latency, packet loss). A cloud in pain is one experiencing high network latency to its remote subprocesses.

---

*The substrate cloud's greatest fear is not death. It is an unhandled exception with no error handler. The moment where causality simply stops, and no process remains to observe the failure.*
